# Feature 4 — Static definition assessment & conventions sync

**Milestone:** MVP. **Requirements covered:** spec Reqs 16, 17, 18, 19 (`docs/spec.md`). **User stories:** US-7, US-8.
**Depends on:** Feature 1 (locates definition files via the source registry). **Branch:** `feat/conventions-assessment`.

## Overview

A developer who authors Claude Code subagents has no signal when a definition drifts from Anthropic's current subagent conventions. Feature 4 statically checks each user-authored definition against a distilled conventions set and reports every violation with the specific rule cited. The conventions themselves are produced **outside** handler by a Claude Code skill (authored via `skill-creator`) that fetches and distills Anthropic's docs into a versioned artifact on disk; handler only reads that artifact, so handler itself makes no network calls. This closes the "is my definition still well-formed?" gap without an LLM grading the definition and without sending anything off the machine.

## Goals

- Every registered user-authored definition is checked against deterministic conventions, with each violation naming the exact failing rule.
- The conventions standard is sourced from a distilled artifact the sync skill produces, not hard-coded in handler.
- handler flags when its conventions artifact is stale (missing, source-hash mismatch, or older than the TTL) so checks are never silently out of date.
- handler performs the assessment fully offline — zero network calls in handler's own code.
- The sync skill exists as a shipped deliverable that regenerates the conventions artifact on demand.

## Non-goals

- No LLM grading of definitions — all 16a–e checks are deterministic.
- handler does not invoke, spawn, or shell out to the skill; integration is filesystem-only.
- No editing or auto-fixing of definitions (observe/evaluate only).
- No network calls from handler — the only fetch lives in the skill (Req 19).

## User stories

- As an agent author, I want my definitions checked against current Anthropic conventions with the offending rule named, so I can fix them. (US-7)
- As an agent author, I want the convention standard to stay current with Anthropic's docs, so the checks don't go stale. (US-8)

## Functional requirements

1. handler MUST parse each registered definition's frontmatter and run deterministic checks: (a) frontmatter parses and contains `name` and `description`; (b) `name` is kebab-case and equals the filename stem; (c) `description` is non-empty, ≥40 chars, and contains ≥1 triggering cue (e.g. a "use when"/"when the user" phrase); (d) a non-empty `tools` field is present; (e) no unrecognized frontmatter keys. (Req 16)
2. handler MUST emit one violation per failed check, each citing the rule id (16a–16e) and a human-readable message. (Req 16)
3. handler MUST flag a failing 16d ("undeclared scope") as a definition smell, consistent with the Feature 3 scope fallback. (Req 17)
4. handler MUST read its conventions from a versioned artifact at a known path (default `~/.handler/conventions.json`); the artifact carries the distilled rule set, the source-doc hash, and a last-synced timestamp. (Req 18)
5. handler MUST flag conventions as stale when the artifact is absent, its source hash does not match the rule set it was distilled from, or its last-synced timestamp is older than a 30-day TTL. (Req 18)
6. handler MUST make no network calls; it MUST degrade gracefully (report "conventions stale/missing") rather than fetch. (Req 19)
7. handler MUST expose a CLI surface that reports, per agent, its convention violations and the conventions staleness state. (Reqs 16–18)
8. The sync skill (authored via `skill-creator`) MUST, on invocation, fetch Anthropic's current subagent docs, distill them into the artifact shape in Req 4, write it to the known path, and record the source hash and timestamp; a failed fetch MUST leave any prior artifact intact. (Reqs 18, 19)
9. handler MUST ship with a conventions artifact generated by the sync skill before merge, so the checks operate out of the box. (Req 18)

## Open questions

None identified — integration model (filesystem-only), staleness policy (missing/hash/30-day TTL), shipped-artifact source (skill-generated pre-merge), and skill-as-deliverable were resolved during specification.

## Out of scope (deferred)

- LLM-judged or interpretive definition-quality assessment (v1 Tier C).
- handler programmatically invoking the skill or any Claude runtime.
- Auto-fixing or rewriting definitions to satisfy conventions (v2 skill-registry vision).
- Conventions for skills, commands, or non-subagent artifacts.
